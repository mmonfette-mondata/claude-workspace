# Understanding Proto Files in md.eco.sg Project

**Date:** 2026-05-26

## Question

I don't understand what proto files are and where and when to use them in the md.eco.sg project.

## Answer

### What are Proto Files?

Proto files (`.proto`) are **interface definition files** written in Protocol Buffers language. They define:
1. **Data structures** (messages) - like JSON schemas, but language-agnostic
2. **API contracts** (services) - like OpenAPI specs, but for gRPC
3. **How data gets serialized** on the wire - much more efficient than JSON

Think of proto files as **the contract between different parts of your system**. They answer: "What shape is the data?" and "What operations are available?"

### Why Do Proto Files Exist? The Core Problem They Solve

Imagine you have:
- A **domain-service** written in Go that stores tenant data
- A **BFF** (Backend-for-Frontend) written in Go that exposes HTTP APIs
- A **worker in another domain** that needs to consume tenant events

**Without proto files:**
```go
// domain-service defines this struct
type Tenant struct {
    MsTenantId string
    DisplayName string
    // ... dozens of fields
}

// BFF manually creates HTTP client code and hardcodes the structure
// Worker manually parses JSON and hopes the structure matches
// When you add a field, you update 3+ places manually
// When you change a field type, things break silently
```

**With proto files:**
```protobuf
// tenant.proto - SINGLE SOURCE OF TRUTH
message Tenant {
  string ms_tenant_id = 1;
  string display_name = 2;
  TenantState state = 3;
  google.protobuf.Timestamp created_at = 4;
  google.protobuf.Timestamp updated_at = 5;
}

service TenantService {
  rpc CreateTenant(CreateTenantRequest) returns (CreateTenantResponse);
  rpc GetTenant(GetTenantRequest) returns (GetTenantResponse);
}
```

Then **code generation tools** automatically create:
- Go structs with serialization logic
- gRPC server interfaces
- gRPC client code
- Type-safe methods

### Real Example from md.eco.sg: The Complete Flow

#### Step 1: Write the Proto Contract

`domains/ms-entities/proto/private/ms_entities/private/v1/tenant_service.proto`:
```protobuf
syntax = "proto3";

package sg.ms_entities.private.v1;

import "ms_entities/private/v1/tenant.proto";
import "ms_entities/private/v1/common.proto";

option go_package = "github.com/elucidia/md.eco.sg/domains/ms-entities/gen/go/private/ms_entities/private/v1;msentitiesv1";

// Intra-domain gRPC service consumed by the BFF.
service TenantService {
  // Creates a new tenant.
  rpc CreateTenant(CreateTenantRequest) returns (CreateTenantResponse);
  // Gets a tenant by MS Tenant ID.
  rpc GetTenant(GetTenantRequest) returns (GetTenantResponse);
  // Lists tenants with pagination, filtering, search, and sorting.
  rpc ListTenants(ListTenantsRequest) returns (ListTenantsResponse);
  // Updates a tenant's display name.
  rpc UpdateTenant(UpdateTenantRequest) returns (UpdateTenantResponse);
  // Transitions a tenant to a target state.
  rpc UpdateTenantState(UpdateTenantStateRequest) returns (UpdateTenantStateResponse);
  // Rotates the tenant's encryption key.
  rpc RotateEncryptionKey(RotateEncryptionKeyRequest) returns (RotateEncryptionKeyResponse);
}

// Request for CreateTenant.
message CreateTenantRequest {
  string ms_tenant_id = 1;
  string display_name = 2;
}

// Response for CreateTenant.
message CreateTenantResponse {
  Tenant tenant = 1;
}

// Request for GetTenant.
message GetTenantRequest {
  string ms_tenant_id = 1;
}

// Response for GetTenant.
message GetTenantResponse {
  Tenant tenant = 1;
}
```

This proto file is:
- **Human-readable** - anyone can understand what the API does
- **Language-agnostic** - could generate code for Go, Python, Java, TypeScript
- **Version-controlled** - changes are tracked in git

#### Step 2: Generate Code with Buf

When you run `make generate`, the **buf** tool reads your proto files and generates Go code:

```bash
# Input: domains/ms-entities/proto/private/.../*.proto
# Output: domains/ms-entities/gen/go/private/.../*.pb.go
```

#### Step 3: Implement the Server (domain-service)

`domains/ms-entities/apps/domain-service/internal/tenant/handler.go`:
```go
package tenant

import (
    "context"
    "errors"

    "github.com/jackc/pgx/v5/pgconn"
    "google.golang.org/grpc/codes"
    "google.golang.org/grpc/status"
    "google.golang.org/protobuf/types/known/timestamppb"

    privpb "github.com/elucidia/md.eco.sg/domains/ms-entities/gen/go/private/ms_entities/private/v1"
    pubpb "github.com/elucidia/md.eco.sg/domains/ms-entities/gen/go/public/ms_entities/public/v1"
    "github.com/elucidia/md.eco.sg/libs/common-go/log"
)

// PrivateHandler implements the private TenantService gRPC interface.
type PrivateHandler struct {
    privpb.UnimplementedTenantServiceServer  // Generated interface
    svc Service
}

// NewPrivateHandler creates a new PrivateHandler.
func NewPrivateHandler(svc Service) *PrivateHandler {
    return &PrivateHandler{svc: svc}
}

// CreateTenant creates a new tenant.
func (h *PrivateHandler) CreateTenant(ctx context.Context, req *privpb.CreateTenantRequest) (*privpb.CreateTenantResponse, error) {
    if req.MsTenantId == "" {
        return nil, status.Error(codes.InvalidArgument, "ms_tenant_id is required")
    }
    if req.DisplayName == "" {
        return nil, status.Error(codes.InvalidArgument, "display_name is required")
    }

    t, err := h.svc.Create(ctx, req.MsTenantId, req.DisplayName)
    if err != nil {
        if isUniqueViolation(err) {
            return nil, status.Errorf(codes.AlreadyExists, "tenant with ms_tenant_id %s already exists", req.MsTenantId)
        }
        log.Ctx(ctx).Err(err).KV("ms_tenant_id", req.MsTenantId).Error("create tenant failed")
        return nil, status.Errorf(codes.Internal, "create tenant: %v", err)
    }

    return &privpb.CreateTenantResponse{Tenant: toPrivateProto(t)}, nil
}

// GetTenant returns a tenant by MS Tenant ID.
func (h *PrivateHandler) GetTenant(ctx context.Context, req *privpb.GetTenantRequest) (*privpb.GetTenantResponse, error) {
    if req.MsTenantId == "" {
        return nil, status.Error(codes.InvalidArgument, "ms_tenant_id is required")
    }

    t, err := h.svc.Get(ctx, req.MsTenantId)
    if err != nil {
        if errors.Is(err, ErrNotFound) {
            return nil, status.Errorf(codes.NotFound, "tenant %s not found", req.MsTenantId)
        }
        log.Ctx(ctx).Err(err).KV("ms_tenant_id", req.MsTenantId).Error("get tenant failed")
        return nil, status.Errorf(codes.Internal, "get tenant: %v", err)
    }

    return &privpb.GetTenantResponse{Tenant: toPrivateProto(t)}, nil
}

// ListTenants lists tenants with pagination and filtering.
func (h *PrivateHandler) ListTenants(ctx context.Context, req *privpb.ListTenantsRequest) (*privpb.ListTenantsResponse, error) {
    params := ListParams{
        PageSize:  int(req.PageSize),
        PageToken: req.PageToken,
        State:     fromPrivateProtoState(req.State),
        Search:    req.Search,
        SortField: fromPrivateProtoSortField(req.SortField),
        SortOrder: fromPrivateProtoSortOrder(req.SortOrder),
    }

    result, err := h.svc.List(ctx, params)
    if err != nil {
        log.Ctx(ctx).Err(err).Error("list tenants failed")
        return nil, status.Errorf(codes.Internal, "list tenants: %v", err)
    }

    var tenants []*privpb.Tenant
    for _, t := range result.Tenants {
        tenants = append(tenants, toPrivateProto(t))
    }

    return &privpb.ListTenantsResponse{
        Tenants:       tenants,
        NextPageToken: result.NextPageToken,
        TotalCount:    int32(result.TotalCount),
    }, nil
}
```

The handler **implements the generated interface** - the compiler enforces that all RPC methods exist with the correct signatures.

#### Step 4: Call from the BFF

`domains/ms-entities/apps/bff/internal/server/server.go`:
```go
package server

import (
    "context"
    "encoding/json"
    "fmt"
    "net/http"
    "strconv"
    "strings"
    "time"

    "google.golang.org/grpc"
    "google.golang.org/grpc/codes"
    "google.golang.org/grpc/credentials/insecure"
    "google.golang.org/grpc/status"

    "github.com/elucidia/md.eco.sg/domains/ms-entities/apps/bff/internal/auth"
    "github.com/elucidia/md.eco.sg/domains/ms-entities/apps/bff/internal/config"

    privpb "github.com/elucidia/md.eco.sg/domains/ms-entities/gen/go/private/ms_entities/private/v1"
    log "github.com/elucidia/md.eco.sg/libs/common-go/log"
)

// tenantJSON is the JSON shape returned for each tenant.
type tenantJSON struct {
    MsTenantID           string `json:"ms_tenant_id"`
    DisplayName          string `json:"display_name"`
    State                string `json:"state"`
    EncryptionKeyVersion int32  `json:"encryption_key_version"`
    CreatedAt            string `json:"created_at"`
    UpdatedAt            string `json:"updated_at"`
}

// stateToString converts a proto TenantState to the lowercase JSON string.
func stateToString(s privpb.TenantState) string {
    switch s {
    case privpb.TenantState_TENANT_STATE_PENDING:
        return "pending"
    case privpb.TenantState_TENANT_STATE_ACTIVE:
        return "active"
    case privpb.TenantState_TENANT_STATE_SUSPENDED:
        return "suspended"
    case privpb.TenantState_TENANT_STATE_DELETED:
        return "deleted"
    default:
        return "unspecified"
    }
}

// parseState maps the HTTP query string value to a TenantState enum.
func parseState(s string) (privpb.TenantState, bool) {
    switch strings.ToLower(s) {
    case "":
        return privpb.TenantState_TENANT_STATE_UNSPECIFIED, true
    case "pending":
        return privpb.TenantState_TENANT_STATE_PENDING, true
    case "active":
        return privpb.TenantState_TENANT_STATE_ACTIVE, true
    case "suspended":
        return privpb.TenantState_TENANT_STATE_SUSPENDED, true
    case "deleted":
        return privpb.TenantState_TENANT_STATE_DELETED, true
    default:
        return privpb.TenantState_TENANT_STATE_UNSPECIFIED, false
    }
}

func toTenantJSON(t *privpb.Tenant) tenantJSON {
    item := tenantJSON{
        MsTenantID:           t.MsTenantId,
        DisplayName:          t.DisplayName,
        State:                stateToString(t.State),
        EncryptionKeyVersion: t.EncryptionKeyVersion,
    }
    if t.CreatedAt != nil {
        item.CreatedAt = t.CreatedAt.AsTime().UTC().Format(time.RFC3339)
    }
    if t.UpdatedAt != nil {
        item.UpdatedAt = t.UpdatedAt.AsTime().UTC().Format(time.RFC3339)
    }
    return item
}

// BFF creates a gRPC client (generated code) and calls the domain-service
func (s *Server) handleListTenants(w http.ResponseWriter, r *http.Request) {
    // Parse HTTP query parameters
    pageSize := parsePageSize(r.URL.Query().Get("page_size"))
    pageToken := r.URL.Query().Get("page_token")
    stateStr := r.URL.Query().Get("state")
    search := r.URL.Query().Get("search")
    sortBy := r.URL.Query().Get("sort_by")
    sortOrder := r.URL.Query().Get("sort_order")

    // Convert to proto enum
    state, ok := parseState(stateStr)
    if !ok {
        http.Error(w, `{"error":{"code":"INVALID_ARGUMENT","message":"invalid state"}}`, http.StatusBadRequest)
        return
    }

    // Call domain-service via gRPC (type-safe!)
    resp, err := s.tenantClient.ListTenants(r.Context(), &privpb.ListTenantsRequest{
        PageSize:   int32(pageSize),
        PageToken:  pageToken,
        State:      state,
        Search:     search,
        SortField:  parseSortField(sortBy),
        SortOrder:  parseSortOrder(sortOrder),
    })
    if err != nil {
        handleGRPCError(w, err)
        return
    }

    // Convert proto response to JSON
    items := make([]tenantJSON, len(resp.Tenants))
    for i, t := range resp.Tenants {
        items[i] = toTenantJSON(t)
    }

    json.NewEncoder(w).Encode(map[string]interface{}{
        "items":           items,
        "next_page_token": resp.NextPageToken,
        "total_count":     resp.TotalCount,
    })
}
```

The BFF doesn't care about gRPC wire protocols - the **generated client handles everything**. The BFF just:
1. Parses HTTP request
2. Calls generated gRPC client with type-safe request objects
3. Converts proto response to JSON

### The Two Categories: Public vs Private

Your project has a brilliant separation:

| **Public Protos** | **Private Protos** |
|-------------------|---------------------|
| Location: `proto/public/` | Location: `proto/private/` |
| **Inter-domain** - other domains can see these | **Intra-domain** - only your domain uses these |
| **Examples:** Lifecycle events, cross-domain services | **Examples:** BFF ↔ domain-service CRUD APIs |
| **Consumers:** Other domains' workers, services | **Consumers:** Your own BFF, your own workers |
| **Change policy:** Breaking changes affect other teams! | **Change policy:** You control all consumers |
| Generated to: `gen/go/public/` | Generated to: `gen/go/private/` |

### Lifecycle Events: Proto Files Powering Cross-Domain Communication

One of the most powerful uses of proto in your project is **FAT Lifecycle Events** (LCEs):

`api/spec/lifecycle/v1/lifecycle.proto`:
```protobuf
syntax = "proto3";

package lifecycle.v1;

import "google/protobuf/field_mask.proto";
import "google/protobuf/timestamp.proto";
import "ms_entities/public/v1/tenant_snapshot.proto";

option go_package = "github.com/elucidia/md.eco.sg/api/go/lifecycle/v1;lifecyclev1";

// Type of lifecycle event.
enum EventType {
  EVENT_TYPE_UNSPECIFIED = 0;
  EVENT_TYPE_CREATED = 1;
  EVENT_TYPE_UPDATED = 2;
  EVENT_TYPE_DELETED = 3;
}

// FAT lifecycle event envelope (ADR-009).
message LifecycleEvent {
  string event_id = 1;
  google.protobuf.Timestamp timestamp = 2;
  EventType event_type = 3;
  google.protobuf.FieldMask changed_fields = 4;
  google.protobuf.FieldMask encrypted_fields = 5;
  oneof entity {
    sg.ms_entities.public.v1.TenantSnapshot tenant = 10;
    // ... other domain entities
  }
}
```

**How it works:**
1. **ms-entities domain**: Creates a tenant → Writes `LifecycleEvent` (serialized as protobuf bytes) to outbox table → Relay publishes to Kafka
2. **notifications domain**: Consumes Kafka message → Deserializes using the generated proto code → Has full tenant snapshot with all fields
3. **No manual JSON parsing**, no version skew, no "field was renamed and broke everything"

The protobuf binary format is stored in the database:

```sql
-- outbox table
CREATE TABLE outbox (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  topic TEXT NOT NULL,
  key TEXT NOT NULL,
  value BYTEA NOT NULL,  -- Protobuf-serialized lifecycle event
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

### When to Use Proto Files in md.eco.sg

Based on the grpc.md guideline and your project structure:

| **Use Case** | **Proto Location** | **When** | **Example** |
|--------------|-------------------|----------|-------------|
| BFF needs to call domain-service | `proto/private/<entity>_service.proto` | Every feature that adds API endpoints | CreateTenant, ListTenants |
| Domain publishes lifecycle events | `proto/public/<entity>_snapshot.proto` + `api/spec/lifecycle/v1/lifecycle.proto` | When other domains need to know about state changes | TenantSnapshot in LifecycleEvent |
| Another domain needs to query your domain | `proto/public/<entity>_service.proto` | Rarely - prefer async events over sync calls | ListActiveTenants for worker sharding |
| Defining what a domain entity looks like | `proto/private/<entity>.proto` and `proto/public/<entity>.proto` | At the start of domain development | tenant.proto with all fields |

### The Deeper Principle: Interface-Driven Design

Proto files embody a fundamental software engineering principle: **Define interfaces before implementation**.

This is similar to:
- **OpenAPI specs** - define HTTP APIs before writing server code
- **TypeScript interfaces** - define shapes before implementing classes
- **Database schemas** - define tables before writing queries
- **Contracts in TDD** - define test cases before writing code

**Benefits:**

1. **Type safety** - The compiler catches mistakes at build time
   - "You forgot to implement GetTenant" → compile error
   - "GetTenant expects string, you passed int" → compile error

2. **Parallel development** - Frontend team can use generated types while backend implements
   - BFF team generates client stubs from proto
   - domain-service team implements server from same proto
   - Both work in parallel, interfaces match by construction

3. **Versioning** - Breaking changes are detected by buf before merging
   - Renaming a field → buf breaking fails
   - Changing a field type → buf breaking fails
   - Removing a field → buf breaking fails

4. **Documentation** - The proto IS the documentation (no drift)
   - Comments in proto become godoc in generated code
   - RPC signatures are self-documenting
   - No "code changed but docs didn't" problem

5. **Efficient serialization** - Protobuf is smaller and faster than JSON
   - 50-70% smaller on the wire
   - 2-5x faster to serialize/deserialize
   - Schema evolution built-in (add fields without breaking old clients)

### Comparing Proto to Alternatives

| Approach | Proto Files | REST + JSON | GraphQL |
|----------|------------|-------------|---------|
| **Wire format** | Binary (Protobuf) | Text (JSON) | Text (JSON) |
| **Size** | ~50% smaller | Large | Medium |
| **Speed** | Fastest | Slower | Slower |
| **Type safety** | Full (generated code) | Manual or codegen | Manual or codegen |
| **Contract** | .proto files | OpenAPI (optional) | Schema (required) |
| **Versioning** | Built-in (buf breaking) | Manual | Additive only |
| **Streaming** | Built-in (server/client/bidirectional) | SSE only | Subscriptions |
| **Code generation** | Multiple languages (Go, TS, Python, Java) | Per-language tools | Per-language tools |
| **Use case** | Microservices, high-perf | Public APIs, simple | API aggregation |

**Why md.eco.sg uses proto:**
- **Internal microservices** - BFF ↔ domain-service, domain ↔ domain communication
- **High throughput** - Kafka lifecycle events (thousands/sec) benefit from compact binary format
- **Type safety** - Catch bugs at compile time, not runtime in production
- **Contract enforcement** - buf breaking detection prevents accidental breaking changes
- **Cross-domain consistency** - All domains use the same LifecycleEvent envelope structure

### Important Rules from grpc.md

1. **Public and private protos never import each other** - They are completely independent module trees
2. **The domain-service implements BOTH interfaces** - Single binary, registers both public and private gRPC services
3. **Only inter-domain protos are published** - Other domains pull only `proto/public/`
4. **LCE envelopes are always inter-domain** - Primary mechanism for cross-domain communication
5. **BFF-consumed services are always intra-domain** - BFF lives inside the domain
6. **Never introduce breaking changes to published packages** - Use buf breaking detection

### Summary

**Proto files are:**
- **Contract definitions** for data structures (messages) and APIs (services)
- **The single source of truth** that code generation builds from
- **Language-agnostic** - same .proto generates Go, TypeScript, Python, Java, etc.
- **Version-controlled** - changes tracked in git alongside code

**In md.eco.sg, you use proto files to:**
1. **Define gRPC services** between BFF and domain-service (`proto/private/`)
2. **Define lifecycle event payloads** for cross-domain communication (`proto/public/` + `api/spec/lifecycle/`)
3. **Define cross-domain query services** when sync calls are needed (`proto/public/`)
4. **Ensure type safety** across service boundaries
5. **Enable parallel development** - teams work with generated stubs before implementation complete

**The workflow is:**
1. Write `.proto` files (the contract) in `proto/public/` or `proto/private/`
2. Run `make generate` (buf generates `.pb.go` and `_grpc.pb.go` files in `gen/go/`)
3. Implement the generated server interfaces (domain-service handler)
4. Use the generated client types (BFF, other domains)
5. Commit both proto AND generated code to git

**Proto files exist because:**
- Writing serialization logic, RPC plumbing, and client/server code by hand is error-prone and time-consuming
- Protocol Buffers + gRPC automates all of it
- You get type-safe, high-performance, version-checked communication
- Single source of truth prevents the "code and docs diverged" problem
- Breaking change detection prevents accidental API breakage

**Key insight:** Proto files are to microservices what OpenAPI specs are to REST APIs, but with automatic code generation, binary efficiency, and built-in versioning.
